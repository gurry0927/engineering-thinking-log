# 工程師思維日誌：撰寫原則 (GUIDE)

為了讓這些紀錄不但能幫助未來的自己，也能更有效地讓 AI 協助你（例如總結、找問題、建議方案），我們遵循以下 5+1 個核心原則。

## 1. 標題要描述「決策情境」
不要只寫技術名詞（如：`Odoo CSS 問題`），要寫出當時在做的判斷。
- **範例**：`判斷 UI 問題責任層級 (Odoo View vs CSS)`

## 2. 紀錄「當時的錯誤假設」
多數人只寫結論，但寫下「我以為...後來發現不是...」能幫助 AI 理解你的思考邊界。
- **為什麼**：這是思考邊界的輪廓，能避免以後走同樣的爛路。

## 3. 明確標示「不可動」與「可實驗」
- **不可動（Constraints）**：原生欄位、Production Data、老闆交代的期限。
- **可實驗（Experimental）**：CSS 樣式、瀏覽器 Override、暫時性的 Patch。
- **效果**：AI 才能知道你是風險厭惡型還是激進型。

## 4. 把「驗證方法」當成一級公民
不是只寫「我試了有效」，而是：
- 我怎麼驗證？
- 驗證失敗代表什麼？
- 驗證成功代表什麼？
- **效果**：這會讓 AI 學到你如何減少不確定性。

## 5. 用「我不確定」當結尾
留下未解的邊界，AI 才不會幫你亂補答案，也會主動針對這些不確定點提供協助。

## 6. 區分「實際行動」與「外部建議」
在對話中，AI 經常會主動提議提供清單或計畫。
- **原則**：如果在對話中沒有明確回覆「要」或「好」，這些內容應歸類為「外部建議」，而非你的實際決策。相關內容應放在日誌最下方的「外部建議與提議 (AI Suggestions)」區塊。
- **效果**：確保日誌真實反映「你的意志與行動」，避免紀錄被 AI 的過度主動引導。
