# 決策紀錄：Vibe Coding 時代下的「邏輯設計員」

## 情境背景
- **系統／專案**：快速交付自動化專案。
- **目標**：利用 AI 快速完成代碼實作，同時保有對系統的控制權。
- **約束條件**：
    - 時間極短（一個早上）。
    - 實作者為技術新手，無法從零手寫複雜架構。

## 我一開始的假設
- **假設 A**：如果是 AI 寫的 Code，那這個專案就算不上是「我的作品」。
- **假設 B**：只要我會下 Prompt，我就算掌握了這門技術。
> **當初為什麼這樣想？** 對於「創作」的定義還停留在傳統的「手刻每一筆」；但在現代開發中，「組裝與編排」才是更高級的創作。

## 感到不安的點 (Risk Assessment)
- **我擔心會發生什麼？** 被資深工程師或是面試官酸「只是會用 AI」，一旦 AI 跑不動我就是廢物。

## 驗證與實驗 (First-class Citizen)
- **實驗操作**：
    - **流程自主權實驗**：刻意不讓 AI 決定流程，而是我自己畫出 Flowchart 後，再讓 AI 填充細節。
    - **UX 優化實驗**：主動加入「去彈窗、載入介面」，這些是 AI 原始建議中沒有的「使用者感知優化」。
- **觀察指標**：
    - 在遇到 Drive API 版本衝突時，AI 無法直接解決。我是否能透過「查閱官方文件 + 分析 Log」找出問題？
- **驗證結果**：成功！Debug 的 4 小時證明了：AI 只是打字機，真正的 Debug 核心是人類對文件的解讀與對問題的定位。

## 決策與行動
- **正式變更**：擁抱「組裝式開發」 (Modular Assembly)。我負責「架構、UX、邊界處理」，AI 負責「語法、樣板」。
- **持續深化**：雖然現在用 AI，但必須透過「看懂別人的 Code (如 Python 舊版)」來強化自己的底層邏輯轉化力。

## 可遷移判斷 (Transferable Skill)
- **判斷原則**：AI 寫出的代碼是廉價的，但「如何串接 A 與 B、如何解決 C 錯誤」的決策過程是昂貴且不容忽視的。
- **提醒未來的自己**：AI 做得出功能，但做不出「體貼」。你的價值在於那個讓使用者感到舒服的「一鍵寄回」。

## 尚未解決的不確定性
- 當 AI 生成的代碼出現深層邏輯 Bug（非語法錯誤）時，我的診斷能力限界在哪裡？
