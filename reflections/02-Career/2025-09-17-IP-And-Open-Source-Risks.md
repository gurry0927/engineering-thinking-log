# 決策紀錄：個人開源作品與公司智財權 (IP) 的風險防線

## 情境背景
- **系統／專案**：將工作中的通用模組（如 GS 轉 Excel 工具）轉化為個人 GitHub 開源作品。
- **目標**：增加作品集，同時確保法律上的絕對安全。
- **約束條件**：
    - 就業合約可能包含「在職創作歸屬」條款。
    - 創作動機起源於公司的實際問題。

## 我一開始的假設
- **假設 A**：只要是我回家寫的、沒用到公司資料，這東西就是我的。
- **假設 B**：AI 寫出的程式碼不需要擔心智慧財產權問題。
> **當初為什麼這樣想？** 忽略了「在職期間產出且與業務相關」這一模糊地帶在法律上的爭議性。

## 感到不安的點 (Risk Assessment)
- **我擔心會發生什麼？** 被公司主張該作品涉及內部機密或專利，甚至因「外流工具」被解除勞動合約。

## 驗證與實驗 (First-class Citizen)
- **實驗操作**：
    - **合約深度檢查**：回頭翻閱所有簽過的聲明，確認 IP 條款。
    - **諮詢模擬實驗**：撰寫「給 HR/主管的詢問信」草稿。
- **觀察指標**：
    - 如果主動報備，公司的態度是支持、冷淡還是嚴禁？
- **驗證結果**：(待填寫)

## 決策與行動
- **最安全行動**：**離職後再搞**（若涉及核心業務邏輯）。
- **在職替代方案**：開發「完全通用」的小工具（如：無關業務的 API 轉換器），並嚴格遵守「個人電腦、個人時間、不使用公司代碼、不使用公司網路」的四大原則。
- **授權選擇**：若開源，優先考慮 MIT 或 Apache-2.0 授權。

## 可遷移判斷 (Transferable Skill)
- **判斷原則**：技術可以分享，但「業務的情境與秘密」絕對不能出校。
- **提醒未來的自己**：不要為了增加一個 GitHub Star，冒著失去整個職涯誠信的風險。

## 尚未解決的不確定性
- 離職後的「競業禁止」或「保密期限」具體年限與範圍。
