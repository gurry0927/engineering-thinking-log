# 決策紀錄：自動化任務是「機器學習」的彎路還是基石？

## 情境背景
- **系統／專案**：職涯路徑與技能樹規劃。
- **目標**：入職「AI 部門」卻一直在做「Google Sheet 自動化」，引發技能生疏的焦慮。
- **約束條件**：
    - 公司目前優先需求是流程自動化。
    - 機器學習 (ML) 專案尚未正式啟動。

## 我一開始的假設
- **假設 A**：不寫模型、不清資料，就是在浪費時間、忘記專業。
- **假設 B**：串接 API 與寫 GAS 這種「小技能」對未來硬核 ML 職位沒有幫助。
> **當初為什麼這樣想？** 因為對 AI 工程師的認知還停留在「實驗室/單機模型訓練」，忽略了企業環境中「系統落地」的複雜性。

## 感到不安的點 (Risk Assessment)
- **我擔心會發生什麼？** 一年後我的模型功力退步，變成只會寫腳本的行政工程師，失去市場競爭力。

## 驗證與實驗 (First-class Citizen)
- **實驗操作**：
    - **技能代換實驗**：將「串 API」視為「ML 模型接入點」練習；將「GAS 自動化」視為「Data Pipeline ETL」練習。
    - **週末微練習進度**：每週花 1 小時在 Kaggle 維持 Data Cleaning 的手感。
- **觀察指標**：
    - 掌握 API 技能後，是否更容易理解 AI 雲端服務（如 OpenAI API）的系統架構？
    - 在開發工具時，是否有意識地運用資料清洗的邏輯（如格式校正規範化）？
- **驗證結果**：(待填寫)

## 決策與行動
- **正式變更**：接受目前的「工程落地先，數據科學後」階段。專心吃下 API 串接與系統整合技能。
- **心態調整**：理解沒落地的模型只是代碼，能跑進產品的模型才是價值。現在練的就是那條「跑進去」的隧道。

## 可遷移判斷 (Transferable Skill)
- **判斷原則**：不要看工具的名子 (GAS/Excel)，要看背後的邏輯 (Data flow/API control)。
- **提醒未來的自己**：如果你只會訓練模型，你只是個科學家；如果你會串 API 落地，你才是真正的 AI 工程師。
