# PRD 與 PM：業務需求與技術實現的橋樑

## 核心概念
在大公司中，開發不再是「隨便寫寫」，而是需要一套標準化的流程來讓不同職能的人（工程師、設計師、老闆）認知對齊。

## 詳細筆記

### 1. PRD (Product Requirement Document)
*   **定義**：產品需求文件，開發流程的核心導航。
*   **關鍵內容**：
    *   **為什麼做？** (Background/Goals)
    *   **做什麼？** (Functional Requirements)
    *   **做到什麼程度？** (Acceptance Criteria)
*   **隱喻**：它是產品的「藍圖」，在動工（寫程式）之前必須確認。

### 2. PM (Product Manager vs Project Manager)
*   **Product Manager (產品經理)**：負責「靈魂與方向」，定義產品要做什麼，並產出 PRD。
*   **Project Manager (專案經理)**：負責「骨架與進度」，協調資源，確保專案準時上線。
*   **注意**：在資源緊繃的環境，PM 常需一人兼顧兩者（既想開發，又管進度）。

## 學習心得
身為工程師，聽懂 PM 的 PRD 是為了確保「我沒寫錯東西」。如果對需求有疑慮，應該在 PRD 階段就提出，而不是寫完程式才發現方向歪了。
