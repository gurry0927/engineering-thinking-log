# 自動化規模化：為什麼從 Sheets 轉向資料庫？

## 📌 背景
在團隊開發初期，Google Sheets 是一個完美的「輕量級資料庫」，具有零成本、高可視化、易於開發的優點。但隨著 P-05 (報價中繼網頁) 的啟動，我們開始面臨 Sheets 的局限性。

## 🔍 Sheets vs. 正式資料庫 (SQL/NoSQL)

| 維度 | Google Sheets (GAS) | 正式資料庫 (e.g., PostgreSQL, Firestore) |
| :--- | :--- | :--- |
| **讀寫速度** | 較慢，且有併發寫入衝突風險 | 極快，支援高併發 |
| **資料容量** | 單表有格數限制 (1000萬格) | 近乎無限擴展 |
| **關係查詢** | 需手動邏輯，效率低 | 支援複雜 SQL Join 或索引查詢 |
| **接口能力** | 主要透過 AppScript Web App | 原生支持 API、ORM 等開發工具 |

## 🏗️ 轉向資料庫的架構思考
1. **資料準確性**：正式資料庫可以設定 Schema 和約束 (Constraints)，防止人工隨意修改造成的資料損毀。
2. **多版本管理**：在報價中繼器中，同一個案號需要儲存多個版本的報價對比，資料庫的關係映射能更優雅地處理這種 1:N 的關係。
3. **前端解耦**：使用資料庫後，WebApp 不需要再依賴 GAS 的 API 轉接，可以直接透過後端 API 獲取數據，效能大幅提升。

## 💡 個人成長點
從「只要能動就好」的 Sheets 操作，進階到思考「資料穩健性與併發處理」的資料庫體系，是邁向後端架構師的重要一步。
