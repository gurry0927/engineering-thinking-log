# 識別碼不等於數字：前導 0 的實務避險策略

## 📌 問題現象
在多數商業系統中，分機號碼、工號、序號鮮少以 `0` 開頭（例如 `0123`）。

## 🔍 背後的工程雷區

### 1. 資料型別的隱性轉換 (Type Coercion)
多數系統（特別是 Excel 或 CSV 匯入器）會下意識將「長得像數字的東西」當成 `Number / Integer`。
- **結果**：`0123` -> 轉存為 `123`。前導 0 永久消失。

### 2. 顯示與儲存的脫節
有些前端會為了好看自動補 0，但底層資料庫存的是無 0 的數值。這在系統串接 (API/Integration) 時會導致 Match 失敗。

### 3. 歷史系統指令衝突
在舊式電話交換機或路由系統中，`0` 常被預留給「撥打外線」或特定的交換指令。

## 🛠️ 專業設計原則

### ✅ 原則一：語意判定 (Semantics Over Look)
只要這個欄位 **不參與數學運算**（加減乘除），即使它全部由 0-9 組成，也應該被定義為 **`String (字串)`**。

### ✅ 原則二：輸入端強心針
在設計 UI 介面（如 Kintone 或 Web Form）時：
- **正確**：使用單行文本、限制長度。
- **錯誤**：使用數值欄位（這會強迫系統進行型別轉換）。

### ✅ 原則三：避險命名
如果系統能自主控制識別碼生成，建議避開 0 開頭，改用：
- `1` 開頭（如 `1001`）
- 英文字母加數字（如 `EXT1001`）
