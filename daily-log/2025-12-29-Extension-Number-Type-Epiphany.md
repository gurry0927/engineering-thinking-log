# 每日開發日誌：2025-12-29 (消失的 0：資料型別的實務修煉)

## 📝 今日事件：關於分機號碼的細微觀察
- **發現**：整理聯絡人資料時，發現幾乎沒有人使用「0」開頭的分機號碼。
- **推論**：這不是巧合，而是實務上為了避開「資料型別轉換」引發的雷區。
- **核心學習**：前導 0 在數值型別 (Integer) 中會被自動消除，導致資料遺失。實務上的默契其實是「長期踩雷後形成的避險習慣」。

## 💡 關鍵決策與邏輯
1. **識別碼 vs 數值**：分機號碼雖然由數字組成，但其語意是「標籤/識別碼」，而非「數量」。
2. **設計原則**：所有標籤性質的欄位（電話、分機、身分證字號），**永遠必須使用 String (字串) 儲存**。
3. **避險設計**：與其教育所有人使用正確型別，不如在設計之初就避開「容易誘發錯誤」的資料內容（如避開 0 開頭）。

## 🚩 遇到挑戰
- **隱性規則的不確定性**：不確定這是默契還是經驗不足。
- **解決方案**：透過邏輯拆解 (Type Conversion / Semantics)，確認這是系統穩定性的基石。

## 💭 今日反思
- **從「使用者」到「結構人」**：開始看見資料背後的「型別博弈」。
- **工程幽靈**：許多看似隨意的規範，背後可能都藏著幾十年前某位工程師熬夜修過的 Bug。
